Executive Summary

Project Overview and Final Outcomes

Purpose of This Report

This final report documents the complete Engine Predictive Maintenance project — a machine learning initiative designed to predict engine failures before they occur. The report covers the full end-to-end pipeline: data registration, exploratory analysis, data preparation, model building with hyperparameter tuning and MLflow tracking, model deployment as a Streamlit web application on Hugging Face Spaces, and an automated GitHub Actions CI/CD pipeline. It is intended for senior decision-makers and technical evaluators assessing the project's business value and technical rigour.

Business Problem

Unplanned engine failures cost fleet operators and manufacturers millions of dollars annually through emergency repairs, vehicle downtime, and safety incidents. Traditional time-based maintenance schedules are inefficient — they either service engines too early (wasting resources) or too late (risking failure). This project builds a data-driven predictive maintenance system that uses real-time engine sensor readings to flag engines at risk of failure before a breakdown occurs, enabling condition-based maintenance at scale.

Key Findings

  • Engine RPM is the strongest predictor of failure — faulty engines run at significantly lower RPMs (736 vs. 885 RPM on average), indicating mechanical stress or degradation.
  • Gradient Boosting achieved the best performance with a Test F1-Score of 76.3% and an 85.2% recall rate, correctly identifying 85 out of every 100 faulty engines.
  • All six ensemble and tree-based models were trained, tuned with GridSearchCV, and tracked via MLflow — enabling full reproducibility and audit trails.
  • The model is deployed as a live Streamlit application on Hugging Face Spaces, enabling real-time fault probability prediction from sensor inputs.
  • An automated GitHub Actions CI/CD pipeline orchestrates the full ML workflow — from data registration through model training to deployment — triggered on every push to the main branch.

Business Impact Potential

Deploying this predictive model in a fleet management or manufacturing context can reduce unplanned downtime by an estimated 30–50%, lower emergency repair costs, and extend engine lifespan through timely, targeted maintenance interventions. The live deployment and automated retraining pipeline ensure the system remains accurate as fleet conditions evolve over time.

3.2 Outlier Treatment – IQR Winsorization

Rather than removing outliers (which would reduce the dataset and potentially discard valid extreme readings), IQR-based Winsorization was applied. Values below Q1 − 1.5×IQR are capped at the lower bound; values above Q3 + 1.5×IQR are capped at the upper bound. This preserves all 19,535 records while eliminating the distorting effect of extreme values.

Section 4: Model Building

4.1 Algorithm Selection Rationale

Six ensemble and tree-based classifiers were selected based on their suitability for tabular sensor data with moderate class imbalance. All models are non-parametric, handle non-linear feature interactions, and do not require feature scaling.

🌟 Gradient Boosting (GBC)
Sequential ensemble that corrects residual errors. Excellent at capturing complex non-linear patterns. Best performer in this study (F1=76.3%).
⚡ XGBoost
Optimized gradient boosting with regularization (L1/L2). Faster training with comparable accuracy to GBC (F1=76.1%).
📈 AdaBoost
Adaptive boosting that focuses on misclassified samples. Robust to noise, interpretable weak learner weights (F1=76.1%).
🏠 Bagging Classifier
Bootstrap aggregation reduces variance. Highest recall (87.5%) — minimizes missed faults at cost of more false alarms (F1=76.1%).
🌳 Random Forest
Parallel ensemble of decorrelated trees. Robust baseline with built-in feature importance (F1=75.8%).
🌳 Decision Tree
Single tree baseline for interpretability comparison. Lowest performance (F1=75.3%) but fully explainable decision rules.

4.2 Training Strategy – GridSearchCV with Stratified K-Fold

Hyperparameter optimization was performed using GridSearchCV with StratifiedKFold(n_splits=5) as the cross-validation strategy. The F1-Score was used as the scoring metric to optimize for fault detection performance under class imbalance.

📄 See Appendix A.2 for the complete GridSearchCV hyperparameter tuning implementation (train.py).
Best Parameters (Gradient Boosting): n_estimators=200, max_depth=5, learning_rate=0.1 — The deeper trees (max_depth=5) and more estimators (200) allow the model to capture complex multi-feature interaction patterns that shallower configurations miss.

8.1 Key Business Insights

📈 Insight 1: RPM is the Early Warning Signal

Engine RPM accounts for 41.2% of the model's predictive power. Engines operating below ~750 RPM under normal load conditions are at significantly elevated fault risk. Monitoring RPM thresholds in real-time provides the earliest possible warning of developing mechanical issues.

⛽ Insight 2: 85% Fault Detection Prevents Breakdowns

The model catches 85.2% of faulty engines before failure. In a fleet of 1,000 engines, this means approximately 852 potential breakdowns are flagged for preventive action. At an average emergency repair cost of $5,000–$15,000 per breakdown, this represents $4.3M–$12.8M in potential annual savings.

🔥 Insight 3: Fuel System Anomalies are Secondary Indicators

Fuel_Pressure contributes 21.4% to model predictions. Elevated fuel pressure in faulty engines suggests injector blockage or fuel pump degradation. Targeted fuel system inspections when both RPM is low AND fuel pressure is elevated can improve maintenance precision.

⚠ Insight 4: Sensor Quality is Critical

The Coolant_Temperature sensor malfunction (195.53°C) demonstrates that sensor data quality directly impacts model reliability. Implementing automated sensor validation checks — flagging readings outside physically plausible ranges — is essential for production deployment.

# train.py — Model training with MLflow tracking (GitHub Actions Job 3) from sklearn.ensemble import GradientBoostingClassifier from sklearn.model_selection import GridSearchCV, StratifiedKFold from sklearn.metrics import f1_score, accuracy_score, roc_auc_score import mlflow, mlflow.sklearn, joblib mlflow.set_tracking_uri("http://localhost:5000") mlflow.set_experiment("engine_predictive_maintenance") cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42) param_grid = {'n_estimators':[100,200], 'max_depth':[3,5], 'learning_rate':[0.05,0.1]} with mlflow.start_run(run_name="GradientBoosting_tuned"): gs = GridSearchCV(GradientBoostingClassifier(random_state=42), param_grid, cv=cv, scoring='f1', n_jobs=-1) gs.fit(X_train, y_train) best = gs.best_estimator_ y_pred = best.predict(X_test) mlflow.log_params(gs.best_params_) mlflow.log_metric("f1_score", f1_score(y_test, y_pred)) mlflow.log_metric("roc_auc", roc_auc_score(y_test, best.predict_proba(X_test)[:,1])) mlflow.sklearn.log_model(best, "model") joblib.dump(best, 'data/best_model.pkl')
