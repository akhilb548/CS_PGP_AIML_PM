# Project Report Guidelines

A strong project report demonstrates not only your technical skills but also your ability to communicate business insights clearly. A good data scientist explains results in a way that both technical and non-technical stakeholders can understand.

The Project Report is where you translate technical findings into meaningful business outcomes. It is not just about showing code or outputs — it is about storytelling with data.

---

## Project Submissions — Expectations

For every project, you are expected to submit two files:

1. **Project Report** (PDF format)
2. **Python Notebook** (converted to HTML format)

Both files are mandatory. Your submission will be graded primarily on the Project Report, while the Jupyter Notebook will be used for verification of the results and code execution.

---

## Interim Submission

The Interim Submission should include the following sections in the report:

- Data Registration
- Exploratory Data Analysis
- Data Preparation
- Model Building with Experimentation Tracking

---

## Final Submission

The Final Submission should include the complete report with the following sections:

- Data Registration
- Exploratory Data Analysis
- Data Preparation
- Model Building with Experimentation Tracking
- Model Deployment
- Automated Github Actions Workflow
- Output Evaluation
- Actionable Insights and Recommendations

---

## What is Expected in a Project Report?

1. Cover all sections in the correct sequence, as per the rubric. Avoid skipping sections.
2. Present a step-by-step logical approach to solving the problem, with reasoning behind every step.
3. Support explanations with tables, graphs, and plots (e.g., histograms, box plots, bar graphs).
4. Provide a clear interpretation of results in simple English that decision-makers can understand.
5. Include Business Insights and Recommendations to connect findings back to the business context.

---

## What an Inadequate Project Report Looks Like

1. Filled with Python code and little to no explanation or interpretation.
2. Contains one-line answers without logic, insights, or explanations.
3. Directly exporting a Jupyter Notebook as a PDF and submitting it as the report.

---

## Dos for Project Report

- Follow all guidelines provided by the Program Office.
- Keep the report professional and concise (ideally within 25 pages).
- Place all raw code and raw outputs in the Appendix, not in the main body.
- Present EDA results in a clean tabular form for clarity and comparison.
- Ensure tables, figures, and charts are properly labeled with legible axis labels, legends, and titles.
- Show all numbers up to 2 decimal places (unless otherwise required).
- Provide a final section of Actionable Insights & Recommendations.
- Use Appendices for supplementary material (raw code, extended tables, additional plots).
- Maintain professional formatting with numbered pages, properly captioned visuals, and consistent style.

---

## Don'ts for Project Report

- Do not rely only on visualizations without modeling — a project must have predictive modeling.
- Do not fill the main body of the report with raw code or outputs.
- Do not copy-paste unreadable tables or chart snapshots directly from Python outputs.
- Do not use non-standard abbreviations or acronyms without explanation.
- Do not submit incomplete reports or skip sections mentioned in the rubric.
